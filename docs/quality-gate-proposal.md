# 2段階品質ゲートモデル 導入提案

## 1. 背景

### 1.1 発端
opencode-mobile プロジェクト（OpenCode Web UI モバイル最適化 UserScript 開発）において、以下の問題が繰り返し発生した。

### 1.2 発生した問題一覧

| # | 問題 | 分類 | 検出タイミング | 検出者 |
|---|------|------|---------------|--------|
| 1 | DOM セレクタが実機と不一致（`[data-action="project-menu"]` は存在しなかった） | 実装ミス | ユーザー実機テスト | ユーザー |
| 2 | ロケール依存セレクタ（`aria-label` が UI 言語で変化し一致しない） | 設計ミス | general worker が自律修正 | general worker |
| 3 | 破壊的 CSS（`#root > * { padding-top: 0 }` が送信ボタンを消失させた） | 実装ミス | ユーザー実機テスト | ユーザー |
| 4 | nav 配置方式の迷走（sticky→fixed→sticky→fixed と 4 回変更） | 設計ミス | ユーザー実機テスト | ユーザー |
| 5 | `node_modules/` 35 万行が git に誤って混入 | 実装ミス | quality-gate | quality-gate agent |
| 6 | handoff ファイルの重複（`handoff.md` + `handoff-20260527.md`） | 設計ミス | ユーザーの質問 | ユーザー |
| 7 | 2 セッション同時起動時の handoff 競合 | 設計ミス | ユーザーの質問 | ユーザー |
| 8 | `resume` コマンドの glob パターンが `handoff.md` 固定で日付付きを見つけられない | 設計ミス | ユーザーの resume 実行 | ユーザー |

### 1.3 問題の構造分析

上記 8 件のうち:

- **設計ミス**: 5 件（#2, #4, #6, #7, #8）→ **5/8 = 62.5% が設計起因**
- **実装ミス**: 3 件（#1, #3, #5）→ 3/8 = 37.5%

現在の quality-gate agent は実装ミス（#1 セレクタ検証, #5 git 管理）をカバーしているが、設計ミスは**完全にノーガード**である。

## 2. 提案: 2 段階品質ゲートモデル

```
要件定義
  │
  ▼
┌─────────────────────────────────────────────┐
│ GATE 1: 設計レビュー                          │
│ 担当: orchestrator (plan フェーズ)             │
│ タイミング: 実装着手前                          │
│                                               │
│ チェック項目:                                   │
│ ・エッジケースマトリクス（後述）                    │
│ ・要件カバレッジ確認（全要件に設計上の回答があるか）    │
│ ・依存関係・前提条件の明示                        │
│ ・既知の制約・リスクの列挙                         │
│                                               │
│ 成果物: 設計書（エッジケース含む）                  │
│ 不合格時: 実装に進めない                          │
└─────────────────────────────────────────────┘
  │ (PASS)
  ▼
実装
  │
  ▼
┌─────────────────────────────────────────────┐
│ GATE 2: コード品質                            │
│ 担当: quality-gate agent                     │
│ タイミング: 実装完了後                           │
│                                               │
│ チェック項目:                                   │
│ ・セレクタ検証（DOM 構造との一致）                  │
│ ・CSS 構文チェック                               │
│ ・JS 構文チェック                                │
│ ・テスト実行・カバレッジ                           │
│ ・.gitignore / git 管理状態                     │
│ ・成果物一覧と実際のファイルの一致                  │
│                                               │
│ 不合格時: blocker 登録 + 修正タスク生成            │
└─────────────────────────────────────────────┘
  │ (PASS)
  ▼
完了
```

### 2.1 既存との差分

| 項目 | 現行 | 提案 |
|------|------|------|
| GATE 1（設計） | orchestrator の plan はタスク分解のみ。設計レビューなし | **設計レビューを必須化**。エッジケースマトリクスを成果物に含める |
| GATE 2（コード） | quality-gate がコード品質をチェック | **変更なし**（現行で十分機能している） |
| ゲート強制 | なし。dispatcher は任意で quality-gate を呼び出す | **dispatcher が GATE 1 → GATE 2 の通過を強制**。未通過時は次工程に進めない |

## 3. GATE 1 の詳細設計

### 3.1 orchestrator の plan フェーズ拡張

現在の orchestrator の `plan` モードの出力に、以下の必須セクションを追加する:

```markdown
## エッジケースマトリクス

| 観点 | 想定される問題 | 設計上の対策 | 検証方法 |
|------|---------------|-------------|---------|
| ブラウザ差異 | Firefox/Chrome でイベント挙動が異なる | onclick インライン方式 | 両ブラウザで手動テスト |
| ロケール | aria-label が UI 言語で変化 | data-component 属性を使用 | selector-validate テスト |
| 同時実行 | 2 セッションが同じ handoff を読む | .done リネーム + ガード | 設計レビュー |
| バージョン互換 | OpenCode のアップデートで DOM 変化 | セレクタを疎結合に | 実機確認 |
| ... | ... | ... | ... |

## 要件カバレッジマトリクス

| 要件ID | 要件 | 設計上の回答 | カバー状況 |
|--------|------|------------|-----------|
| FR-01 | ビューポート高さ安定化 | CSS: h-dvh → 100% | ✅ |
| FR-02 | 不要パディング除去 | CSS: padding: 0 | ✅ |
| FR-03 | セッション一覧視認性 | タップ64px, 自動スクロール | ✅ |
| ... | ... | ... | ... |
```

### 3.2 エッジケース観点の標準テンプレート

orchestrator が常に考慮すべき観点:

1. **実行環境の差異**: ブラウザ（Chrome/Firefox/Safari）、OS（Windows/Android/iOS）
2. **ロケール・言語**: UI テキストの多言語対応、aria-label の変動
3. **同時実行・競合**: マルチセッション、ファイルロック、リソース共有
4. **バージョン互換性**: 対象アプリのアップデートによる DOM/API 変化
5. **初期状態・エッジ値**: データ 0 件時、画面サイズ境界値、認証状態
6. **ネットワーク・接続**: オフライン、Tailscale 未接続、認証失敗
7. **プロセス境界**: handoff ファイルの消費/重複検出、グロブパターン設計
8. **非侵襲性**: 既存 UI への副作用（CSS の過剰な上書き）
9. **可逆性**: スクリプト無効化で完全に元に戻るか

## 4. 実装タスク

### 4.1 orchestrator の agent 定義修正

ファイル: `~/.opencode/agents/orchestrator.md`（または該当の agent 定義ファイル）

修正内容:
1. `plan` モードの出力仕様に「エッジケースマトリクス」「要件カバレッジマトリクス」を必須セクションとして追加
2. エッジケース観点の標準テンプレート（3.2 参照）をチェックリストとして組み込む
3. plan 完了時に `dashboard_add_blocker(type="DESIGN_REVIEW")` を立て、dispatcher が設計承認するまで実装に進めない仕組み

### 4.2 dispatcher の agent 定義修正

ファイル: dispatcher の agent 定義

修正内容:
1. orchestrator の plan 結果を受け取った後、以下を検証:
   - エッジケースマトリクスが存在するか
   - 要件カバレッジマトリクスが存在するか
   - 全要件が「カバー状況: ✅」になっているか
2. 不足がある場合: orchestrator に再 plan を要求
3. 充足している場合: 「設計レビュー PASS」として execute に進む
4. 実装完了後: quality-gate の PASS を確認してから完了とする

### 4.3 quality-gate の変更

**変更不要**。現行の quality-gate はコード品質チェックとして十分機能している。ただし、handoff ファイルの整合性チェック（重複、ガード、行数正確性）をチェック項目に追加することを推奨。

### 4.4 common-rules.md の整備

現在 `C:\Users\nakam\.opencode\shared\common-rules.md` が存在しない。以下を最低限含める:
- dispatcher の関門強制ルール（GATE 1 → GATE 2 の通過を必須化）
- 状況確認の優先順位（プロセス → Git → 実ファイル → ログ → ダッシュボード）

## 5. 期待される効果

### 5.1 予防できる問題（opencode-mobile の事例に基づく）

| 今回の問題 | GATE 1 で予防 | GATE 2 で予防 |
|-----------|:---:|:---:|
| DOM セレクタ不一致 | - | ✅ selector-validate |
| ロケール依存セレクタ | ✅ エッジケース観点「ロケール」 | - |
| 破壊的 CSS（送信ボタン消失） | ✅ エッジケース観点「非侵襲性」 | - |
| nav 配置方式の迷走 | ✅ エッジケース観点「実行環境の差異」 | - |
| node_modules git 混入 | - | ✅ .gitignore チェック |
| handoff 重複 | ✅ エッジケース観点「プロセス境界」 | - |
| 2 セッション競合 | ✅ エッジケース観点「同時実行・競合」 | - |
| resume glob 不一致 | ✅ エッジケース観点「プロセス境界」 | - |

**8 件中 5 件（62.5%）が GATE 1 で予防可能**。残り 3 件は GATE 2 がカバー。

### 5.2 定性的効果

- ユーザー（人間）が実機テストでバグを発見する回数が大幅に減少
- 実装の手戻り（sticky→fixed の往復）が削減
- 「とりあえず実装してから考える」→「設計で潰してから実装する」への文化転換
